# 开发流程

## 改动 → 测试 → 提交

1. 修改代码后，push 前必须先按 [TESTING.md](TESTING.md) 跑一遍本地 Docker 全流程测试。
2. 如果测试需要修改 `.env`（新增变量、改配置等），先向用户提示要改什么、为什么改，不要擅自改。
3. 测试通过后才能提交，测试失败要先排查修复，不要带着失败的测试去提交。
4. 提交前，分别检查暂存区和未暂存的改动，判断哪些属于本次改动、哪些不属于（比如临时调试代码、无关的格式化、遗留的测试产物等）。
5. 想清楚暂存区该怎么处理（哪些文件该 add、哪些该拆成单独一次提交、哪些不该提交）之后，向用户说明打算怎么处理，得到确认后再执行 `git add` / `git commit`。
6. 涉及不同关注点的改动（比如一次是功能改动、一次是文档）优先拆成多次提交，不要混在一起。

## Commit message 规范

遵循 Conventional Commits：`<type>: <一句话描述>`，正文可选，说明"为什么"而非"改了什么"。

常用 type：`feat`（新功能）、`fix`（修 bug）、`docs`（文档）、`chore`（杂项/配置）、`ci`（CI/CD）、`refactor`（重构，不改变行为）、`perf`（性能优化）、`test`（测试相关）、`style`（格式/空白等不影响逻辑的改动）、`build`（构建系统或依赖变更）。
